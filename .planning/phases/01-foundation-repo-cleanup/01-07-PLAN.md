<objective>
Add a single GitHub Actions workflow that runs lint + test + build on every PR to `main` and every push to `main`, using a matrix over both workspaces. The workflow gates merges on success (enforced via branch protection rules by the developer after first run). Env dummies are supplied for the app's Zod validator so build doesn't fail on missing real credentials.

Purpose: FOUND-04 acceptance — CI green required for merge, red CI blocks merge. Closes the feedback loop: any future regression of `no-console` rule (Plan 06), failing Vitest (Plan 04), or broken env validator (Plan 03) fails CI immediately.
Output: One commit `ci: add GitHub Actions lint+test+build workflow`. Plus one checkpoint (`checkpoint:human-action` — developer enables branch protection + adds `SENTRY_AUTH_TOKEN` secret) — without these the workflow runs but the "red CI blocks merge" acceptance is not fully enforced.
</objective>

    name: CI

    on:
      pull_request:
        branches: [main]

    jobs:
      test:
        runs-on: ubuntu-latest
        strategy:
          fail-fast: false
          matrix:
            app: [landing, app]
        steps:
          - uses: actions/checkout@v4

          - uses: actions/setup-node@v4
            with:
              node-version: '20'
              cache: 'npm'
              cache-dependency-path: package-lock.json

          - run: npm ci

          - name: Lint
            run: npm run lint --workspace=apps/${{ matrix.app }}

          - name: Test
            run: npm run test --workspace=apps/${{ matrix.app }}

          - name: Build
            run: npm run build --workspace=apps/${{ matrix.app }}
            env:
              SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}
              NEXT_PUBLIC_SUPABASE_URL: 'https://dummy.supabase.co'
              NEXT_PUBLIC_SUPABASE_ANON_KEY: 'dummy-key-for-build-check-only-20plus-chars'
              NEXT_PUBLIC_SENTRY_DSN: 'https://public@sentry.io/123'

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → repo secrets | `SENTRY_AUTH_TOKEN` exposed as env var to Build step only |
| CI build → public action versions | pinned `actions/checkout@v4` and `actions/setup-node@v4` major versions |
| CI → npm registry | `npm ci` fetches from registry; `package-lock.json` integrity hashes defend against swap |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-07-01 | Information Disclosure | `SENTRY_AUTH_TOKEN` leak via logs | mitigate | GitHub masks secrets in log output by default. Never `echo $SENTRY_AUTH_TOKEN`. Scope is `project:releases` only (not full account) — minimum privilege. |
| T-1-07-02 | Tampering | Malicious PR modifies ci.yml to exfiltrate secrets | mitigate | Branch protection requires approved PR; `pull_request` trigger from forks uses `GITHUB_TOKEN` with restricted scope — secrets NOT exposed to fork PRs by default. For solo-dev repo: this is a low-risk scenario. |
| T-1-07-03 | Denial of Service | Missing branch protection → red CI merged anyway | accept | Checkpoint Task 2 asks developer to enable; if skipped, red CI is visible but merge proceeds. Acceptable solo-dev tradeoff. |
| T-1-07-04 | Supply Chain | `actions/checkout@v4` gets a malicious tag push | mitigate | Using major-version pin `@v4` (not `@main`). GitHub's action integrity scan + Dependabot can raise alerts. Consider SHA-pinning in Phase 2 if risk tolerance tightens. |
| T-1-07-05 | Tampering | `npm ci` pulls compromised transitive dep | accept | `package-lock.json` hashes protect; npm audit runs available; lockfile changes require PR review. Free-tier exposure acceptable for waitlist-stage product. |
</threat_model>
