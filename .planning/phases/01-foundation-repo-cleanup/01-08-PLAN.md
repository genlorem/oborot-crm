| registrar → DNS root | nameserver change is the single irreversible-within-hours step; premature change causes downtime |
| CF Pages build → GitHub Pages deploy | build env vars must match landing's Zod schema; empty = builds succeed but runtime fails |
| CF edge → Sentry | public DSN acceptable; Sentry inbound filters protect (D-08) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-08-01 | Denial of Service | Nameserver change before CF Pages live → prod outage up to 48h | mitigate | Runbook order enforced (Task 1 first, Task 2 only after Task 1 `approved`). Resume-signal gates the sequence. |
| T-1-08-02 | Tampering | Stale Railway still serving oborotcrm.ru after CF cutover → split-brain | mitigate | Task 3 prerequisite confirms Railway service deleted BEFORE committing railway.json removal. |
| T-1-08-03 | Information Disclosure | CF Pages env vars (TG token, Sentry DSN) visible to anyone with CF Pages project access | mitigate | Limit CF project members to the owner + Claude-owned service account (if created). Use separate tokens per environment (Phase 2 can rotate). |
| T-1-08-04 | Tampering | Malicious PR causes CF preview build with attacker env vars | accept | CF preview builds use PRODUCTION env vars by default for this project (set Production+Preview). Fork PRs do NOT receive CF Pages preview by default (GitHub App scope protects). Low risk for solo-dev. |
| T-1-08-05 | Denial of Service | Propagation takes longer than SLA → extended downtime | accept | Rollback plan documented in RESEARCH §Rollout Order — switch nameservers back at registrar. Railway kept alive during cutover (Task 3 step A checks before deleting). |
